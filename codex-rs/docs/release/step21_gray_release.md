# Step 21: 灰度发布与观测驱动的放量

里程碑注记（2025-08-08）
- Serena 集成评估任务阶段 A 完成，已具备原生 in-process 工具路径灰度与放量条件；LSP/Agent 路径将按本步骤策略推进放量与回退演练。

进度概览（摘录）
- 工具路径（In-process）：≈85%，可进入更大范围 Beta 放量
- LSP 路径：≈60%，优先接线 LspFacade 的只读请求（诊断/补全），写路径配合缓冲与防抖
- Agent 执行：≈40%，先完成 A/B 测试框架与回退通道，再择机放量
- 观测与治理、上游同步：≈90%，可支撑灰度与自动回退

未完成与放量前置条件
- 工具 schema/错误码对齐的回归用例补齐
- LSP Facade 对接与文本缓冲策略验证（高频场景）
- Agent 接管路径的安全阈值与指标门槛在 CI/bench 建立

本方案落地“发布节奏 + 观测与回退”，可被集成到任意 CI/CD 与监控体系。

目标
- 将新特性按 Alpha → Beta → 默认开启的节奏推进。
- 全程以指标为导向，出现回归自动回退与告警，保留手动一键回退能力。
- 明确问题记录与修复窗口承诺，支撑快速迭代。

一、发布节奏
1) Alpha（内部使用）
- 受众：内部团队、开发环境。
- 日志：强制开启详细日志（LOG_LEVEL=debug/trace），采集请求/错误/耗时的结构化日志。
- 开关：feature_flag: { name: <feature>, stage: alpha, enabled_for: internal_only }。
- 验收：功能正确性、关键路径覆盖、基础性能无异常。

2) Beta（高级用户与 CI）
- 受众：受邀高级用户、预发布环境、CI 验证任务。
- 观测：以指标为主（错误率、P95/P99 延迟、资源消耗、业务成功率）。
- 反馈：收集外部反馈（Issue/表单/埋点），与指标联动做放量或回退。
- 放量：按人口或流量分层（5%→10%→25%→50%）。每个梯度观察窗口>= 1 个发布周期或满足稳定性条件。

3) 默认开启（保留短期回退）
- 条件：Beta 指标达到基线±阈值内，关键问题关闭。
- 开关：默认on，同时保留一键Kill-Switch（全量关闭）与阶段性回退（逐级缩量）。
- 期限：默认开启后至少保留2个版本的回退能力（保留旧路径/兼容参数）。

二、观测与回退
1) 指标体系（建议）
- 可用性：请求错误率（5xx/总请求）
- 性能：P95/P99 延迟、队列等待时间
- 资源：CPU/内存/GC暂停/连接数
- 业务：转化率/成功率/弃用率/关键漏斗
- 正确性：幂等/一致性校验失败率

2) 基线与阈值
- 基线：从当前稳定版本采集7-14天的分位数与方差，形成参考区间。
- 回归阈值（示例）：
  - 错误率 > 基线均值+3σ 且 持续5分钟
  - P99 延迟 > 基线P99×1.2 且 持续10分钟
  - 业务成功率 < 基线P50×0.98 且 持续15分钟

3) 自动回退
- 触发条件：满足任一回归告警的“严重”级别时，触发回滚流水线或关闭特性开关。
- 策略：优先“缩量”→“关闭Beta”→“回退到Alpha/禁用”。
- 容错：提供冷却时间（例如 15 分钟）避免震荡；启用护栏（最小流量门限）。

4) 告警分级与路由
- 告警级别：Warning（观察）/ Critical（自动回退）
- 路由：值班频道、负责人、On-call 轮值；自动创建工单/Issue。

5) 日志与追踪
- Alpha 阶段强制 debug/trace。Beta 阶段按采样率（1~10%）保留 trace，并对失败/慢请求100%采样。

三、执行清单
- 在 configs/feature_flags/ 中为该特性创建开关配置。
- 在 ops/alerts/ 中启用 Prometheus 告警规则并连接回退脚本/流水线。
- 在 ops/runbooks/ 中登记回退手册与修复窗口承诺。
- 在 CI 中添加 Beta 验证作业（可选，见文末建议）。

四、放量建议
- 首日：5%（1发布周期）
- 稳定后：10% → 25%（各观察1周期）
- 高稳定：50%（2周期）
- 达标：100% 默认开启，同时保留回退能力2个版本

五、与CI的集成（建议）
- 在预发布环境对关键指标做合格线守门（例如错误率≤1%，P95≤目标）。
- CI 内发版后拉取 Prometheus 快照检查，未达标自动打回。
