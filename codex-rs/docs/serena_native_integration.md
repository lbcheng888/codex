# Serena 原生融合 codex-rs 方案（Step 1：明确目标与成功指标）

本文件用于对齐“以原生方式将 serena-agent、serena-lsp、serena-tools 深度融合进 codex-rs 的 agentic coding 流水线”的目标、成功指标、测量方法、验收标准、风险与缓解、实施里程碑，以及开放问题。作为后续设计与实现的上位约束与核验依据。

---

## 一、目标与范围

- 核心目标
  - 在 codex-rs 的 agentic coding 流水线中，以原生（in-process、无跨进程序列化）的方式深度融合 serena-agent、serena-lsp、serena-tools。
  - 完全避免使用 `core/src/serena_inprocess.rs` 与 Serena MCP 服务器路径（含 MCP 协议栈与多进程/IPC 调度）。
  - 保持 codex-rs 现有对话、工具生态与用户可见行为对用户透明（不改变 CLI 与常用交互），同时带来稳定体验提升。

- 非目标
  - 不再维护或增强 MCP 路径与旧进程内管理器。
  - 不改变用户可见的 CLI 命令与基本行为，除非文档明确说明。

- 范围界定
  - 覆盖：serena-agent 的工具路由、计划执行与上下文管理；serena-lsp 的诊断、补全、代码操作等关键请求；serena-tools 的本地工具适配与调用链。
  - 不涉及：模型推理策略本身的重大改变（除非为消除性能瓶颈的必要微调且不影响行为一致性）。

---

## 二、成功指标（SLO/SLA 定义与统计方法）

- 工具调用开销（本地函数/trait 调用链）
  - 指标：p50 ≤ 2ms，p95 ≤ 10ms；零或一次拷贝；严禁跨进程序列化。
  - 采样对象：通过 codex-rs 工具路由触发的每一次 serena-tools 调用（I/O 轻量与纯计算均计）。
  - 度量方法：在统一工具调用入口与实现端注入高精度计时（`std::time::Instant`），记录开始-结束差，带调用 ID 与工具名；落盘至环形内存缓冲并周期性 flush 至 metrics 后端。滚动窗口 5 分钟，输出 p50/p95。

- LSP 关键请求延迟
  - 指标：p95 ≤ 30ms（诊断、补全、代码操作三类分别统计）。
  - 采样对象：serena-lsp 在 codex-rs 内的 in-process 调用，全链路计时（message parse → handler → response），不含 UI 渲染。
  - 度量方法：在 LSP 请求入口与响应出口加埋点，按方法分类统计 p50/p95。诊断合并批次采用固定 debounce 窗口（如 50ms）再计时，避免抖动放大。

- 端到端编程任务耗时
  - 指标：较 MCP 或旧进程内管理器路径总耗时降低 ≥ 25%。
  - 采样对象：标准化 E2E 工作流（示例：打开项目 → 获取诊断 → 补全突发 → 代码操作修复 → 生成补丁 → 运行测试 → 汇总）。
  - 度量方法：定义 `session_start`/`session_end` 与阶段里程碑事件，统计场景中位数与 p95，与基线对比。

- 内存额外占用
  - 指标：新增原生路径相对基线的额外 RSS/heap < 150MB；核心数据零拷贝或一次拷贝。
  - 度量方法：在典型 E2E 运行期，通过 jemalloc/TCMalloc 统计或 `/proc/self/smaps`、RSS 采样，叠加 Rust allocator 事件；对关键数据结构（LSP 文本、AST、工具输入/输出 buffer）采用 `Arc<[u8]>`、`Bytes` 等零拷贝容器并审计复制点。

- 通过度与稳定性
  - 指标：全量 E2E 场景通过率不低于当前稳定版本；崩溃率/致命错误不高于当前；回退开关可即时生效且默认安全（异常时可自动回退）。
  - 度量方法：CI 全量用例、夜间回归、8 小时稳压统计失败率；运行时心跳中检测异常阈值触发自动回退与告警。

## 三、一致性与行为约束

- 对话与工具生态透明：工具名、参数语义、错误分类、重试策略与当前一致；保持接口幂等与错误码兼容。
- LSP 行为一致：诊断 severities、补全排序与过滤规则、代码操作适用范围与冲突策略维持现状或显式改进，并做 A/B 核验。
- 数据语义不变：跨模块传递的数据结构使用相同 schema 与字段命名；版本化变更提供 shim 或保持向后兼容。

## 四、架构与实现原则

- In-process Only：以 crate 方式嵌入 serena-agent、serena-lsp、serena-tools，同进程同内存空间，无 IPC/JSON-RPC 序列化。
- 零/一次拷贝通路：共享只读数据使用 `Arc`/`Bytes`；可变数据采用 `Cow` 或 arena/rope；避免多层字符串拼接与多次 materialize。
- 无锁或细粒度锁：热点路径采用无锁结构或 `parking_lot` 的 `RwLock` 优化；批处理与缓存降低抖动与锁竞争。
- 可观测性内建：统一 metrics trait，编译期 feature gate；生产默认启用新路径，异常可动态回退。
- Feature Flag 与回退
  - 新路径 feature 默认开启（如 `serena_native`）。
  - 运行时提供环境变量/配置键可强制回退。
  - 异常率超阈值、性能倒挂或一致性失败时自动回退并上报。

## 五、测评与基线建立

- 基线版本
  - MCP 路径与旧进程内管理器在 main 上的稳定 commit 作为 baseline，冻结并记录指标。

- 测试场景集（建议最少集）
  - 小型 Rust 项目：open → initial diagnostics → completion bursts → code action（rename/fix）→ run tests
  - 中型 TS/JS 项目：同上 + 大文件补全、跨文件引用跳转
  - 多语言混合仓：Rust + Python + TS，交替触发 LSP 请求与工具调用
  - 高负载压力：并发补全 N=8、诊断频率 20Hz，持续 30 分钟

- 环境约束
  - CPU/内存/磁盘与 CI 一致；Rust stable 版本锁定；禁用节能模式；关闭无关后台进程。

- 统计方法
  - 预热剔除（首 60s 或首 3 次场景）
  - 每场景重复运行 ≥ 30 次，取中位与 p95；进行显著性检验，确认 ≥ 25% 提升。

## 七、风险与缓解

- 性能倒挂
  - 若 LSP 内部数据复制不可避免，采用 arena/rope 优化与 `Bytes`/`Arc` 共享；为大对象提供借用生命周期接口。
- 行为偏差
  - 补全排序与诊断时序可能变化，提供可配置排序权重与诊断 debounce 调优，A/B 验收。
- 并发与死锁
  - 采用 `parking_lot` 并缩短临界区；热点路径设计无锁队列；在测试配置启用死锁检测。
- 记忆泄漏
  - 引入 jemalloc heap profiling 与 leak detector（测试标志），在稳定压测场景下观测。
- 回退失败
  - 回退路径与新路径解耦，触发条件单一且可观测，优先级判定简单化。

## 九、开放问题（需由负责人确认）

1) 最小 Rust 版本与依赖策略：是否允许引入 `bytes`、`parking_lot`、`once_cell` 等依赖？是否要求 no_std 可选？
2) codex-rs 工具总线的统一入口 trait/模块定位（例如 `ToolInvoker`/`ToolRouter` 具体路径）以便统一埋点与零/一次拷贝改造？
3) serena-lsp 在 codex-rs 的当前接入形态：是否仍经由 JSON-RPC 编解码？是否已有内部方法可直调与文本缓冲借用接口？
4) 端到端基线脚本：是否已有标准化脚本与固定工作集？若无，是否接受新增基准脚本并纳入 CI？
5) 回退策略：是否需要运行期动态切换（无重启）？触发与告警通道的期望是什么？
6) 内存预算参照：与 MCP 路径相比、旧进程内管理器相比，或二者均对比？
7) 一致性偏差容忍度：补全列表顺序变化、诊断时序的允许阈值（如前后 10%）？是否存在必须完全一致的关键用例？
